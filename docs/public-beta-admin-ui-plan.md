# 공개 베타 및 관리자 UI 기획 초안

## 목적

이 문서는 현재 내부용 데모 UI와 별개로, 공개 베타용 사용자 화면과
운영자용 관리자 화면을 분리하기 위한 1차 기획 초안이다.

지금 단계의 목표는 백엔드를 전면 재설계하는 것이 아니라, 외부 베타에
내보낼 수 있는 안전하고 이해하기 쉬운 화면 구조를 정의하는 것이다.

## 배경

현재 `/demo/emotion-analysis` 페이지는 내부 점검용으로는 유용하지만,
다음과 같은 내부 개념이 그대로 노출되어 있다.

- raw websocket 이벤트
- `session_id`
- 수동 종료 버튼
- 토큰 입력 기반 연결
- 디버그 성격의 로그 및 JSON 노출

이런 구조는 개발과 QA에는 적합하지만, 일반 대중 대상 공개 베타에는
맞지 않는다.

## 기본 방향

화면을 아래 3개로 분리한다.

- 공개 베타 페이지: 일반 사용자가 실제로 사용하는 화면
- 관리자 페이지: 운영자가 세션과 분석 결과를 모니터링하는 화면
- 내부 데모 페이지: 개발/QA가 계속 사용하는 점검용 화면

## 목표

- 일반 사용자가 기술 지식 없이도 서비스를 이해하고 사용할 수 있게 한다.
- 운영에 필요한 정보는 관리자 화면에서 충분히 볼 수 있게 한다.
- 공개 화면에서 디버그 요소와 인증 우회 요소를 제거해 보안 위험을 낮춘다.
- 기존 세션/분석 카드 백엔드 흐름은 가능한 범위에서 재사용한다.

## 비목표

- 이번 단계에서 현재 내부 데모 페이지를 대체하지 않는다.
- 이번 단계에서 분석 카드 스키마 자체를 새로 설계하지 않는다.
- 이번 단계에서 полноцен한 고객지원 시스템까지 만들지 않는다.
- 관리자 권한 체계를 과도하게 복잡하게 만들지 않는다.

## 화면 정의

### 1. 공개 베타 페이지

주 사용자:

- 일반 베타 사용자

주 역할:

- 서비스 소개 확인
- 로그인 후 대화 시작
- 감정 대화 진행
- 세션 종료 후 결과 확인

반드시 포함할 것:

- 서비스 소개 문구
- 로그인 기반 진입
- 단순한 채팅 중심 레이아웃
- 현재 상태 안내
- 사용자 친화적 로딩 및 오류 메시지
- 세션 종료 후 결과 카드
- 개인정보/주의사항 안내
- 위기 또는 긴급 상황 안내 문구

반드시 제외할 것:

- raw 이벤트 로그
- websocket 이벤트 이름
- `session_id`
- raw JSON payload
- 액세스 토큰 직접 입력
- 수동 디버그 제어 버튼

### 2. 관리자 페이지

주 사용자:

- 마스터 관리자 또는 운영자

주 역할:

- 진행 중인 세션과 종료된 세션 확인
- 세션별 실시간 대화 내역 확인
- 요약 카드 및 분석 카드 확인
- 위험 신호 또는 실패 케이스 빠르게 식별

반드시 포함할 것:

- 세션 목록
- 상태별 필터
- 세션 상세 페이지
- `session_id` 기준 조회
- 실시간 대화 내역 보기
- 최신 요약 카드 및 카드 이력 보기
- 위험도 표시
- 분석 카드 생성 실패 표시
- 검색 기능
- 운영 메모 영역

차후 추가 검토 항목:

- 세션 강제 새로고침
- 카드 재생성
- 감사 로그를 남기는 복사/내보내기
- 운영 후속조치 상태 관리

MVP에서 제외할 것:

- 확인 절차 없는 파괴적 액션
- 모든 관리자에게 동일한 쓰기 권한 부여

### 3. 내부 데모 페이지

주 사용자:

- 개발자
- QA

역할:

- 현재의 websocket 단위 점검 기능 유지
- 디버그/검수 목적 화면으로 유지

원칙:

- 공개 베타 동선과 분리
- 운영 환경에서는 비활성화하거나 관리자 권한 뒤로 숨김

## MVP 범위

### 공개 베타 MVP

- 진입 페이지
- 로그인 후 채팅 페이지
- 한 번에 하나의 활성 세션 화면
- 세션 종료 후 결과 카드
- 기본 재시도/재연결 처리
- 모바일 대응

### 관리자 MVP

- 관리자 인증 또는 권한 체크
- 세션 목록
- 세션 상세
- 대화 내역 보기
- 최신 분석 카드 보기
- `진행 중 / 종료 / 실패 / 고위험` 필터

## 보안 및 운영 요구사항

### 공개 베타

- UI에서 토큰 직접 입력 금지
- 공개 플로우에서 쿼리스트링 토큰 전달 금지
- 운영 환경에서 `EMOTION_NO_AUTH_WEB_TEST` 사용 금지
- raw payload 노출 금지
- 기본적인 rate limit 및 abuse 대응 필요
- 에러 로깅 및 모니터링 필요

### 관리자

- 별도 관리자 권한 체크 필요
- 세션 열람 이력 기록 필요
- 기본은 읽기 전용 권한
- 민감 액션은 별도 확인 절차 필요
- 필요 시 개인정보 마스킹 전략 적용

### 내부 데모

- 운영 환경 기본 비활성화 또는 관리자 보호 하위로 이동

## UX 원칙

### 공개 베타

- 차분하고 단순할 것
- 내부 용어를 쓰지 않을 것
- 한 번에 하나의 핵심 행동만 요구할 것
- 기술 설명보다 신뢰 형성 문구를 우선할 것
- 세션 종료 후 결과 상태를 명확히 보여줄 것

### 관리자

- 정보량은 많아도 읽기 쉬울 것
- 상태 중심으로 빠르게 스캔 가능할 것
- 위험/실패 케이스가 눈에 잘 띌 것
- 대화와 카드가 한 흐름으로 이해되게 할 것

## 데이터 및 재사용 방향

- 기존 websocket 기반 대화 흐름은 재사용한다.
- 저장된 transcript를 기본 원천 데이터로 유지한다.
- 기존 분석 카드 조회 API는 가능한 범위에서 재사용한다.
- 관리자 목록/필터에 필요한 집계 API는 부족한 경우만 추가한다.

## 권장 진행 순서

### 1단계

- 현재 데모 페이지의 운영 노출 방식 정리
- 공개 베타/관리자 화면 범위 확정
- 인증 및 권한 경계 정의

### 2단계

- 공개 베타 페이지 MVP 구현
- 기존 대화 흐름과 연결
- 디버그 요소 제거

### 3단계

- 관리자 세션 목록 및 상세 구현
- 대화 내역과 분석 카드 노출
- 위험/실패 필터 추가

### 4단계

- 운영 보조 기능 확장
- 감사 로그, 메모, 후속조치 기능 보완

## 열린 질문

- 공개 베타 사용자의 로그인 방식은 무엇으로 확정할 것인가?
- 관리자 권한은 기존 사용자 모델에 붙일 것인가, 별도 역할로 둘 것인가?
- 공개 결과 페이지는 최신 카드만 보여줄 것인가, 카드 이력도 보여줄 것인가?
- 운영자가 열람 가능한 대화 범위는 어디까지 허용할 것인가?
- MVP 단계에서 관리자 액션은 읽기 전용이면 충분한가?
