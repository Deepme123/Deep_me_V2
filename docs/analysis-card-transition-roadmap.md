# Analysis Card Transition Backend Roadmap

## 목적

- Analysis Card 전환 시점의 서버 정책을 먼저 고정한다.
- 종료 판단의 source of truth를 단계 추적이 아니라 서버의 마무리 의도와 사용자 확인으로 둔다.
- 이후 구현 중 다시 해석해야 하는 정책 결정을 이 문서에 모은다.

## 핵심 원칙

- `step_order`, `step_type`, 턴 수, 특정 단계 도달 여부는 종료를 위한 보조 신호일 뿐이다.
- 세션 종료의 기준은 `server close intent`와 사용자 응답(`confirm_close` 또는 `cancel_close`)이다.
- 분석카드는 `close_ok` 이후에만 생성한다.
- 서버가 마무리 의도를 냈더라도 사용자가 확인하기 전에는 세션을 종료하지 않는다.

## 종료 판단 모델

서버는 각 assistant 응답 직전에 아래 둘 중 하나의 마무리 의도를 가질 수 있다.

### soft trigger

의미:
- 지금까지의 대화만으로도 유의미한 분석카드를 만들 수 있다.
- 여기서 마무리를 제안하는 것이 자연스럽지만, 사용자가 더 말하고 싶다면 이어가도 된다.

판정 조건:
- `emotion_label`, `topic`, `trigger_summary`, `insight_summary` 중 핵심 필드가 대부분 채워졌다.
- 직전 응답이 이미 요약 또는 정리 성격을 띠고 있다.
- 다음 질문이 새로운 탐색보다 반복 확인에 가까워질 가능성이 높다.
- 사용자가 "이 정도면 된 것 같아", "조금 정리된 것 같아"처럼 느슨한 마무리 신호를 보냈다.

서버 동작:
- 이번 턴의 assistant 응답 끝에 고정 마무리 멘트를 붙인다.
- 같은 턴에 `suggest_close` 이벤트를 보낸다.
- 사용자가 거절하면 대화를 이어간다.

### hard trigger

의미:
- 서버가 보기에 이번 턴 이후 대화를 더 끄는 것이 가치가 낮거나, 사용자가 명시적으로 마무리를 원한다.
- 그래도 강제 종료하지는 않으며, 마지막 확인은 반드시 사용자에게 받는다.

판정 조건:
- 사용자가 명시적으로 종료, 정리, 분석카드 생성을 요청했다.
- 최근 두 번 이상의 user turn에서 실질적인 새 정보 없이 같은 내용이 반복됐다.
- 서버가 이미 카드 생성에 필요한 핵심 요약을 확보했고, 다음 질문이 사실상 filler가 된다.
- 보조 신호로 단계 후반부, 최대 턴 근접, 마무리 단계 도달이 감지됐다.

서버 동작:
- 이번 턴의 assistant 응답 끝에 고정 마무리 멘트를 붙인다.
- 같은 턴에 `suggest_close` 이벤트를 보낸다.
- 같은 응답 안에서 새로운 탐색 질문은 더 붙이지 않는다.

## 고정 마무리 멘트

서버가 `soft trigger` 또는 `hard trigger`를 결정한 턴에는 아래 문구를 그대로 사용한다.

`여기까지 이야기해줘서 고마워. 여기서 대화를 마무리하고 분석 카드를 만들어도 될까?`

규칙:
- soft/hard trigger 모두 같은 문구를 쓴다.
- 문구는 assistant 응답의 마지막에 위치한다.
- 이 문구 뒤에 다른 질문을 붙이지 않는다.
- 문구를 상황별로 변형하지 않는다. 문장 고정이 우선이다.

## 이벤트 의미

### `suggest_close`

- 서버가 이번 턴에 마무리 의도를 냈다는 뜻이다.
- 세션은 아직 열린 상태다.
- 클라이언트는 확인 UI를 보여줄 수 있지만, 아직 분석카드를 만들면 안 된다.

### `confirm_close`

- 사용자가 대화 종료와 분석카드 전환에 동의했다는 뜻이다.
- 서버는 세션 종료 처리를 진행하고, 성공 시 `close_ok`를 보낸다.
- 이 이벤트 이후 같은 라이브 세션으로 일반 대화 메시지를 더 받지 않는다.

### `cancel_close`

- 사용자가 이번 마무리 제안을 거절하고 대화를 더 이어가겠다는 뜻이다.
- 세션은 계속 열린다.
- 현재 close intent는 해제되고, 서버는 재제안 쿨다운에 들어간다.

### `close_ok`

- 서버가 종료 확인을 정상 반영했고 세션을 최종화했다는 뜻이다.
- 이 이벤트가 분석카드 생성의 유일한 시작점이다.
- `close_ok` 자체는 카드 생성 완료를 의미하지 않는다.

## 현재 프로토콜과의 대응

- WebSocket의 기존 `type="close"` 요청은 마이그레이션 기간 동안 `confirm_close`의 legacy alias로 취급할 수 있다.
- 새 정책의 정식 의미 이름은 `confirm_close`, `cancel_close`, `close_ok`다.
- `cancel_close`는 별도 이벤트로 추가한다.

## `cancel_close` 이후 재제안 쿨다운 정책

- 기본 쿨다운은 `2`개의 user turn이다.
- 쿨다운 동안 서버는 자동 `suggest_close`를 다시 보내지 않는다.
- soft trigger와 hard trigger 모두 쿨다운 규칙을 따른다.
- 단, 사용자가 직접 "이제 끝낼래", "분석 카드 만들어줘"처럼 명시적으로 종료를 다시 요청하면 쿨다운을 즉시 무시할 수 있다.
- 쿨다운이 끝난 뒤에도 같은 대화 내용만 반복 중이면 한 턴에 한 번만 다시 제안한다.

## `close_ok` 이후 분석카드 생성 순서

1. 서버가 `confirm_close`를 받는다.
2. 서버가 세션의 최종 요약 필드와 종료 시각을 DB에 반영한다.
3. 서버가 해당 세션을 더 이상 대화 입력을 받지 않는 finalized 상태로 본다.
4. 서버가 `close_ok`를 응답한다.
5. 서버가 `session_id`를 idempotency key로 사용해 분석카드 생성 작업을 시작한다.
6. 분석카드 생성 로직은 메모리상의 임시 상태가 아니라 DB에 저장된 최종 transcript와 summary만 읽는다.
7. 카드 생성이 성공하면 결과를 저장한다.
8. 카드 준비 완료 또는 실패 상태를 클라이언트가 조회하거나 구독할 수 있게 노출한다.

정책:
- `close_ok` 이전에는 어떤 형태로도 자동 카드 생성을 시작하지 않는다.
- 같은 세션에 대해 자동 생성은 한 번만 성공해야 한다.
- 중복 실행이 들어와도 `session_id` 기준으로 dedupe 또는 upsert되어야 한다.

## 실패 시 재시도 정책

- 자동 재시도는 최초 실행 포함 최대 `3`회까지 허용한다.
- 백오프는 `5초`, `30초` 순으로 둔다.
- 재시도 대상:
  - LLM timeout
  - provider `429`
  - provider `5xx`
  - 일시적 네트워크 장애
  - 일시적 DB 오류
  - 모델 출력 파싱 실패 또는 스키마 검증 실패 1회
- 재시도 비대상:
  - 세션이 finalized 되지 않음
  - 필수 transcript 또는 summary 데이터 누락
  - 권한 오류
  - 이미 같은 세션의 성공 카드가 존재함

재시도 실패 후 처리:
- 상태를 `analysis_card_failed`로 남긴다.
- 수동 재시도는 허용하되, 원본 세션 transcript를 다시 열지 않는다.
- 수동 재시도도 같은 `session_id`를 기준으로 중복 카드 생성을 막아야 한다.

## 구현 순서

1. `suggest_close`와 고정 마무리 멘트를 기준으로 close intent를 정리한다.
2. `confirm_close`와 `cancel_close` 이벤트 계약을 추가한다.
3. `cancel_close` 쿨다운 상태를 세션 단위로 관리한다.
4. `close_ok` 이후 분석카드 생성 워크플로를 붙인다.
5. idempotency, 재시도, 실패 상태 노출을 추가한다.

## 완료 기준

- 구현 도중 다시 판단해야 할 서버 정책이 이 문서에 모두 있다.
- 종료 기준이 단계 추적이 아니라 서버 마무리 의도와 사용자 확인으로 명시돼 있다.
- `close_ok` 이후에만 분석카드 생성이 시작되도록 순서가 고정돼 있다.
