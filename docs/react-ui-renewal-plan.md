# React 웹 UI 리뉴얼 계획

## 문서 목적

이 문서는 현재 프로젝트 구조를 기준으로, 기존 QA용 HTML 데모는 유지하면서
제품용 웹 UI를 React로 확장하는 방향을 빠르게 참고할 수 있도록 정리한 계획서입니다.

핵심 목표는 다음과 같습니다.

- 기존 대화 및 분석 카드 테스트 흐름을 깨지 않는다.
- 일반 베타 사용자용 UI와 운영자용 모니터링 UI를 React로 분리해 만든다.
- 백엔드 API와 WebSocket 흐름은 최대한 재사용한다.
- 한 번에 전체 교체하지 않고 점진적으로 리뉴얼한다.

## 현재 전제

현재 저장소에는 이미 다음과 같은 구조가 준비되어 있습니다.

- QA용 데모 UI: `app/backend/demo_ui`
- 제품용 프론트엔드 작업 공간: `frontend/`
- 베타 사용자용 영역: `frontend/apps/beta`
- 관리자용 영역: `frontend/apps/admin`
- 공통 프론트엔드 코드 영역: `frontend/shared`

기존 QA 데모는 다음 경로에서 계속 사용합니다.

- 라우트: `/demo/emotion-analysis`
- 용도: WebSocket 대화 흐름 점검, 분석 카드 회귀 테스트, 내부 QA

즉, 이번 리뉴얼은 기존 HTML 데모를 없애는 작업이 아니라, 제품용 UI를 별도로 키우는 작업입니다.

## 왜 React로 가는가

현재 범위는 단일 HTML 페이지 수준을 넘어가고 있습니다.

- 일반 베타 사용자 대화 화면
- 세션 종료 후 결과 화면
- 운영자 세션 목록 화면
- 운영자 세션 상세 모니터링 화면
- QA용 내부 데모 유지

이 정도 범위가 되면 화면 수, 상태 관리, 라우팅, 공통 컴포넌트, 재사용 로직이 늘어나기 때문에
순수 HTML/JS를 계속 늘리는 방식보다 React 구조가 유지보수에 유리합니다.

다만 QA용 도구 화면은 이미 목적이 분명하고, 회귀 테스트 허브 역할을 하고 있으므로
굳이 React로 다시 만들지 않고 유지하는 편이 더 효율적입니다.

## 리뉴얼 원칙

이번 리뉴얼은 아래 원칙을 따릅니다.

1. 기존 QA 데모는 유지한다.
2. 제품용 UI만 React로 새로 만든다.
3. 백엔드 API와 WebSocket은 최대한 그대로 쓴다.
4. 운영자 UI도 같은 프론트엔드 작업 공간 안에서 함께 설계한다.
5. 랜딩 페이지보다 실제 사용자 흐름 화면을 먼저 만든다.

## 화면 경계

### 1. QA 데모

- 경로: `/demo/emotion-analysis`
- 기술: 기존 HTML/CSS/JS 유지
- 역할:
  - WebSocket 메시지 흐름 확인
  - 세션 종료 흐름 확인
  - 분석 카드 생성 결과 확인
  - 내부 개발 및 QA용 테스트 허브

### 2. 베타 사용자 UI

- 작업 위치: `frontend/apps/beta`
- 예상 라우트:
  - `/beta`
  - `/beta/chat`
  - `/beta/result/:sessionId`

역할:

- 일반 베타 사용자가 실제로 사용하는 화면
- 감정 대화 진행
- 세션 종료 후 결과 확인
- QA용 세부 디버그 정보는 숨김

### 3. 관리자 UI

- 작업 위치: `frontend/apps/admin`
- 예상 라우트:
  - `/admin`
  - `/admin/sessions`
  - `/admin/sessions/:sessionId`

역할:

- 운영자 또는 마스터 사용자가 세션을 모니터링
- 세션 목록 조회
- 상태 필터링
- 대화 내역 확인
- 최신 분석 카드와 실패 상태 확인

## 백엔드 재사용 전략

프론트엔드 리뉴얼의 핵심은 백엔드를 다시 만드는 것이 아니라, 이미 있는 흐름을 프론트에서 안정적으로 재사용하는 것입니다.

우선 재사용 대상은 다음과 같습니다.

- WebSocket: `/ws/emotion`
- 헬스체크: `/health`, `/health/db`
- 세션 및 대화 관련 기존 API
- 분석 카드 조회 관련 기존 API

관리자 목록 화면에서 필요한 집계나 필터 API가 부족할 경우에만 최소 범위로 추가합니다.

## 프론트엔드 구조 제안

권장 구조는 아래와 같습니다.

```text
frontend/
  apps/
    beta/
    admin/
  shared/
```

각 영역의 책임은 다음과 같습니다.

- `frontend/apps/beta`
  - 일반 사용자 화면
  - 대화 플로우
  - 결과 페이지

- `frontend/apps/admin`
  - 운영자 목록 화면
  - 세션 상세 화면
  - 위험 신호 및 상태 확인 UI

- `frontend/shared`
  - API 호출 유틸
  - WebSocket 클라이언트
  - 공통 타입
  - 포맷터
  - 공통 UI 컴포넌트

## 구현 우선순위

React 리뉴얼은 아래 순서로 진행하는 것이 가장 효율적입니다.

### 1단계: 프론트엔드 기본 세팅

- `frontend` 작업 환경 준비
- 공통 타입과 API/WS 유틸 정리
- 베타/관리자 앱의 기본 폴더 구조 정리

목표:

- 이후 화면 작업이 반복되지 않도록 기반을 먼저 만든다.

### 2단계: `beta/chat`

가장 먼저 만들어야 할 핵심 화면입니다.

포함 내용:

- 사용자 입력
- AI 응답 표시
- 연결 상태 표시
- 세션 종료 흐름
- 기본 오류 처리

목표:

- 현재 HTML 데모의 가장 중요한 사용자 플로우를 React에서 재현한다.

### 3단계: `beta/result/:sessionId`

세션 종료 후 결과 화면입니다.

포함 내용:

- 분석 카드 조회
- 결과 요약 표시
- 실패 상태 또는 로딩 상태 표시

목표:

- 대화 종료 후 사용자 경험을 완성한다.

### 4단계: `admin/sessions`

운영자가 전체 세션을 확인하는 목록 화면입니다.

포함 내용:

- 세션 목록
- 상태 필터
- 위험 신호 표시
- 최근 세션 확인

목표:

- 운영자가 전체 흐름을 빠르게 파악할 수 있게 한다.

### 5단계: `admin/sessions/:sessionId`

운영 상세 모니터링 화면입니다.

포함 내용:

- 세션별 대화 로그
- 최신 분석 카드
- 실패 여부
- 필요한 경우 추가 메모 영역

목표:

- 운영과 QA가 실제로 확인해야 하는 정보를 한 곳에 모은다.

### 6단계: `/beta`

마지막에 진입 화면을 정리합니다.

포함 내용:

- 서비스 안내
- 시작 버튼
- 기본 주의 문구

목표:

- 기능이 갖춰진 뒤 진입 경험을 다듬는다.

## 개발 방식

초기 개발 단계에서는 백엔드와 프론트를 분리 실행하는 방식이 가장 편합니다.

- 백엔드: `uvicorn app.main:app --reload`
- 프론트: React 개발 서버 별도 실행

이 방식의 장점:

- 기존 백엔드와 QA 데모를 건드리지 않고 개발 가능
- React 화면을 빠르게 반복 개발 가능
- 배포 전략을 나중에 결정할 수 있음

## 배포 전략

지금 단계에서는 배포보다 UI 구조와 흐름 정리가 우선입니다.

다만 장기적으로는 두 가지 방식 중 하나를 선택할 수 있습니다.

1. 프론트와 백엔드를 분리 배포
2. React 빌드 결과물을 FastAPI가 정적 서빙

초기에는 로컬 개발 기준으로 진행하고, UI 구조가 안정화된 뒤 배포 방식을 정하는 것이 좋습니다.

## 기대 효과

이 계획대로 진행하면 다음 장점이 있습니다.

- 기존 QA 도구를 유지한 채 제품 UI를 확장할 수 있다.
- 대화와 분석 카드 테스트 흐름이 끊기지 않는다.
- 베타 사용자용 화면과 운영자 화면의 역할이 명확해진다.
- 화면 수가 늘어나도 구조가 덜 꼬인다.
- 이후 React 기반의 기능 확장이 쉬워진다.

## 요약

한 줄로 정리하면 다음과 같습니다.

기존 `/demo/emotion-analysis` 는 내부 QA용으로 유지하고, 제품용 웹 UI는 `frontend/` 아래에서 React로 새로 구축한다.

실행 순서는 다음이 가장 현실적입니다.

1. 프론트엔드 기본 세팅
2. `beta/chat`
3. `beta/result/:sessionId`
4. `admin/sessions`
5. `admin/sessions/:sessionId`
6. `/beta` 진입 화면 정리

이 방식이 현재 프로젝트 구조와 앞으로의 확장 방향을 가장 자연스럽게 연결합니다.
